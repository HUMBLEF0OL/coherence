# Skill: coherence:enable-sidecars
Trigger: `/coherence:enable-sidecars`
BRD: FR-COMMANDS-7 · DD-050

## Purpose
Provision `.claude/coherence/sidecars/<name>.yaml` fallback files for each tracked skill and agent. Used when `/coherence:doctor` reports `frontmatter_preserves_unknown_keys: false` — i.e., the installed Claude Code version does not carry coherence metadata through frontmatter, so a separate sidecar file is needed to store anchor, layer, and tracking state per doc. Idempotent: safe to re-run.

## Steps

### 1. Pre-flight check
- Load `.claude/coherence/host-caps.json`.
- If `frontmatter_preserves_unknown_keys: true`: print the following and exit:
  ```
  ℹ  Frontmatter key preservation is available in your Claude Code version.
     Sidecars are not required. Coherence metadata is stored in frontmatter directly.
     No files created.
  ```
- Load `.claude/coherence/config.yaml` to get the `tracked` doc list.

### 2. Identify targets
For each entry in `tracked`:
- If `layer` is `skill` or `agent`: the doc is a candidate for a sidecar.
- Derive sidecar filename: replace `/` with `__` in the workspace-relative path, strip leading `.claude/`, strip extension, append `.yaml`.
  - Example: `.claude/skills/coherence-review/SKILL.md` → `skills__coherence-review__SKILL.yaml`
- Check whether `.claude/coherence/sidecars/<filename>` already exists.

### 3. Report plan
```
─── Enable Sidecars ──────────────────────────────────────
Docs requiring sidecars: <N>
Already provisioned:     <N>
To create:               <N>
─────────────────────────────────────────────────────────
```
If `To create` = 0, print "All sidecars already provisioned." and exit.

List each file to be created. Ask:
```
Create <N> sidecar file(s)? [y/n]
```
Abort if `n`.

### 4. Create sidecar directory
Ensure `.claude/coherence/sidecars/` exists.

### 5. Provision sidecar files (idempotent)
For each target doc that does not already have a sidecar, create:

```yaml
# Coherence sidecar — auto-generated by /coherence:enable-sidecars
# Source: <workspace-relative path to tracked doc>
schema_version: "1.0"
source_doc: "<workspace-relative path>"
layer: "<skill | agent | referring>"
coherence_anchor: "<primary heading slug of the doc>"
last_synced: null
drift_state: clean
tracked_since: "<ISO-8601 UTC now>"
```

Write atomically (temp + rename). (NFR-RELIABILITY-1)

### 6. Update config
In `.claude/coherence/config.yaml`, for each entry in `tracked` that now has a sidecar, add/update field `sidecar: "<relative path to sidecar file>"`. Write config atomically.

### 7. Confirm
```
─── Sidecars enabled ────────────────────────────────────
Created:   <N> sidecar file(s) in .claude/coherence/sidecars/
Updated:   .claude/coherence/config.yaml (sidecar references added)
─────────────────────────────────────────────────────────
Coherence will now use sidecar files to store metadata for skills and agents.
```

## Constraints
- Idempotent: never overwrite an existing sidecar — skip files that already exist. (FR-COMMANDS-7 "idempotent")
- All writes atomic (temp + rename). (NFR-RELIABILITY-1)
- Do not modify the tracked doc files themselves — only create files in `.claude/coherence/sidecars/`. (FR-PERMISSION-1 — no auto-writes to user files in observe mode)
- Do not invoke LLM subagents. (DD-008)
- Only creates sidecars for `skill` and `agent` layer docs; `referring` layer docs (CLAUDE.md, ARCHITECTURE.md etc.) use frontmatter directly when available, or a separate sidecar path if needed.
