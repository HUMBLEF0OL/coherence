## Theme
**v0.1 reacts. v0.2 proposes. v0.3 distributes. v0.4 polishes. v1.0 trusts.**
v1.0 is the **trust + intelligence** release. It graduates coherence from a tool that always asks to one that earns the right to act — through a per-section trust ledger, team-aggregated signals, and invariant assertions that protect sections from patches that violate declared constraints. It also closes the observability gap with a quality metrics command and a deep audit that can reason across sections, not just report on them.
---
## Problem statement
v0.4 polished the install path and locked the trust model (DD-065) in place for the first generation of marketplace installers. Four classes of problem remain:
1. **Trust is binary and stateless.** Observe / Graduated mode is a per-session, per-developer toggle with no memory of past interactions. A section that has been accepted 20 times without revert gets the same review friction as a section seen for the first time. The trust model cannot learn.
2. **Net-new files require explicit human action forever.** DD-065 requires `/coherence:propose-accept` for every net-new skill, agent, or command file regardless of trust history. There is no pathway from "always ask" to "this author has earned auto-land" — even after years of clean signal.
3. **No section invariants.** Sections can declare `coherence:section` anchors but cannot declare what must remain true about their content. A patch that removes the only code example in a section, or that references a function that doesn't exist, passes all current validation gates.
4. **Observability is raw JSONL.** `metrics.jsonl` contains 90 days of events but there is no command that turns them into actionable insight: which sections drift most, which patches get reverted, where cost is concentrated. The `/coherence:audit` command bundles existing commands but does not reason about quality.

## Non-goals (post-v1.0)
- Hosted backend, database, or network upload service — ever (DD-117 permanent).
- Cross-organisation trust federation — sharing trust signals across different repos or orgs.
- AI-generated assertion suggestions — coherence does not propose `asserts:` values; that is author intent.
- Automatic threshold tuning from field data — DD-116 field-calibration execution remains trigger-contract only.
- GUI / web dashboard for metrics — file-only plugin; a web surface would require a backend.
---
## Load-bearing premises (carry-forward)
- **DD-117 — No backend, ever.** Trust ledger, team aggregate, and metrics are all file-only. `coherence/trust-aggregate.json` uses git as the substrate.
- **DD-118 — No legacy version support.** v1.0 installs fresh. No migrator from v0.4 state layout. `.claude/coherence/` per-project state is preserved across re-install (trust ledger survives).
- **DD-065 trust model amended, not abandoned.** The net-new file gate *relaxes* after a trust threshold — it does not disappear. The threshold and the relaxation mechanism are OQ-v1-02 and OQ-v1-09.
- **DD-130 autogen stubs carry forward.** All v1.0 slash commands appear in `plugin.json` and get autogenerated stubs at build time.

## Sequencing gates (v0.4 → v1.0)
**All must be ✅ before v1.0 spec freeze.**
1. ✅ Trust score formula decided → DD-138 (exponential decay normalized ratio, 30-day half-life).
2. ✅ Net-new file trust threshold decided → DD-139 (score ≥ 0.85, ≥ 5 sections, ≥ 30 days → hint + `/coherence:trust --promote`).
3. ✅ Codebase-linked assertion types scoped → DD-141 (`symbol_exists` + `file_exists` in v1.0).
4. ✅ `asserts:` violation policy decided → DD-143 (per-assertion `(block)`/`(warn)`, default warn).
5. ✅ Team trust aggregate conflict resolution decided → DD-140 (recency-weighted per-author mean).
