BRD-2 — FUNCTIONAL REQUIREMENTS (v1.0). Grouped by goal G-1..G-6. All FRs trace to DD-131..DD-147.
Trust ladder (G-1)
- FR-TRUST-1 — Per-section trust score computed via DD-138 formula (weighted exponential decay normalized ratio; α = 0.977/day; 30-day half-life; cap 200 events; div-by-zero guard returns 0.0 when denominator \< 0.001). Score ∈ \[-1.0, +1.0\]. Stored as rolling summary in trust-ledger.json. (DD-131, DD-138)
- FR-TRUST-2 — When personal section score ≥ 0.85, modifying patches auto-apply without confirmation. Additive patches retain v0.2 Graduated auto-apply. Destructive and frontmatter patches always require confirmation regardless of score. (DD-131 audit pass 1 amendment)
- FR-TRUST-3 — /coherence:trust command surface. Subcommands: --status (print current trust state + promote eligibility), sync (write own author-hash file to coherence/trust/), --promote --auto-land \<kinds\> (activate auto-land for declared kinds; default annotate only), --prune-stale (remove files \> 365 days).
- FR-TRUST-4 — Promote eligibility hint emitted when ALL three conditions met: (a) personal score ≥ 0.85 on ≥ 5 distinct sectionRefs with current score \> 0.0, (b) ledger spans ≥ 30 days from first event, (c) hint not previously emitted. One-time CLI hint; user runs --promote to activate. (DD-139)
Cross-session learning (G-2)
- FR-LEDGER-1 — Personal trust ledger at .claude/coherence/trust-ledger.json. Gitignored, per-developer. Atomic writes (write-temp-then-rename). Survives plugin re-install per DD-118.
- FR-LEDGER-2 — Team aggregate as per-developer files at coherence/trust/\<author-hash\>.json (committed). Author hash = 12-hex SHA-256 of git config user.email. Each developer writes only their own file via /coherence:trust sync. No merge conflicts possible.
- FR-LEDGER-3 — Each per-developer file carries last_synced_at ISO timestamp. Files older than 180 days excluded from aggregate computation at /coherence:metrics read time (still on disk; returning developer reactivates on next sync). /coherence:trust --prune-stale physically removes files \> 365 days. (DD-132 audit pass 3 amendment)
- FR-LEDGER-4 — Team aggregate score per section = arithmetic mean of each active developer's personal trust score (DD-138 formula) for that section. Computed at /coherence:metrics read time — no aggregate cache. Conflict (\~0 score) signals contested section. (DD-140)
asserts: frontmatter (G-3)
- FR-ASSERTS-1 — Sections may declare asserts: in frontmatter as a YAML list of objects with type (required), param (optional), policy (optional, default warn). Parsed by existing parseAnchors.ts using js-yaml. Max 10 assertions per section; excess logged and ignored. (DD-143 audit pass 2 amendment)
- FR-ASSERTS-2 — Text-pattern assertion types (always-on, no LLM, regex/string-based): has_example, no_placeholder_links, max_words:N, min_words:N, no_todo_comments. Run synchronously in validation pipeline before Stage 2. (DD-133 audit pass 1 amendment)
- FR-ASSERTS-3 — Codebase-linked assertion types: symbol_exists, file_exists. Reuse language file-glob and grep patterns from hallucination registry via new assertSymbolExists() wrapper. param accepts optional language suffix (e.g. pythonFunc:python); supported tokens: typescript, javascript, python, go, rust. Omitted suffix falls back to detectProjectLanguage(). (DD-141)
- FR-ASSERTS-4 — Policy enforcement: policy: block rejects the patch with violation message; policy: warn (default) proceeds with \[assertion warning\] notice in review UX. Per-assertion granularity. (DD-143)
Quality metrics (G-4)
- FR-METRICS-1 — /coherence:metrics renders 5 Markdown sections to chat: (1) Summary all-time + 30-day accepts/reverts/edits/discards, (2) Top drifting sections by applied-patch count from coherence-log.md, (3) Trust scores top-10 high and top-10 low from trust-ledger.json, (4) Cost trend 30-day Unicode sparkline, (5) Revert hotspots with rate \> threshold (default 20%, min 5 events). (DD-147)
- FR-METRICS-2 — Flags: --since \<ISO-date\> filters time range (reuses export-metrics filtering logic). --out \<path\> writes report file (DD-128 sandbox rules apply). --revert-threshold \<pct\> overrides default hotspot threshold. Sections with insufficient data show graceful 'No data yet' messages.
- FR-METRICS-3 — Revert rate formula: reverts / (accepts + reverts + edits). All event types in denominator because edits represent partial disagreement. (DD-147 audit pass 1 amendment)
Deep audit (G-5)
- FR-AUDIT-2 (supersedes v0.4 FR-AUDIT-1 placeholder) — /coherence:audit free tier always-on: token budget analysis per section (Normal \< 2000, Large 2000–5000 with advisory, Bloated \> 5000 with strong warning); preserves v0.4 bundling (doctor + scope-debug + status + metrics-export). No LLM calls. \< 100 ms p95. (DD-135)
- FR-AUDIT-3 — --deep flag adds LLM cross-section consistency pass. Targets section pairs that share ≥ 1 symbol from section-symbol-index.json. Hard cap 10 pairs per invocation. Bounded token cost \~\$0.05 worst case at Sonnet pricing. (DD-142)
- FR-AUDIT-4 — Cost gate before LLM call: prints estimated cost and pair count; requires explicit y/N confirmation. Pairs \> 10 → user advised to narrow with --sections \<glob\>. Mirrors --estimate confirmation pattern from Stage 1.
- FR-AUDIT-5 — section-symbol-index.json (gitignored) built lazily on first --deep invocation. Invalidated by SHA-256 hash of section-index.json (not mtime — unreliable on Windows/network FS). Cache hit O(1); miss O(sections × symbols) bounded by 10-pair cap early termination. (DD-142 audit pass 2 amendment)
Trust signals (G-6)
- FR-SIGN-1 — Release tarball signed via cosign keyless using GitHub Actions OIDC token. CI-only; no key management. Rekor transparency log entry produced for independent verification. (DD-145)
- FR-SIGN-2 — Local release script (scripts/release-ga.mjs) exits non-zero when GITHUB_ACTIONS env absent at sign step. --unsigned flag explicitly required to bypass; output file renamed to cohrence-\<version\>-UNSIGNED.tgz to prevent accidental distribution. (DD-145 audit pass 2 amendment)
- FR-SIGN-3 — SHA-256 manifest at release-artifacts/cohrence-\<version\>.sha256 (committed for git provenance) + printed in GitHub Release notes body. release-artifacts/ added to .npmignore; M-LEGACY-1 gate verifies absent from tarball.
- FR-SIGN-4 — SECURITY.md at project root (GitHub/GHSA/Dependabot standard location). Documents responsible disclosure path.
- FR-SIGN-5 — README adds Verification section: links to release-artifacts/, cosign verify command line, Rekor lookup, and M6 static-analysis gate descriptions as verifiable claims.
Repair operations (cross-cutting)
- FR-REPAIR-1 — /coherence:repair extended: detects orphaned trust-ledger.json keys (present in ledger but absent from section-index.json). Lists orphaned entries; offers --reassociate "old-ref":"new-ref" flag or --expire-orphans bulk auto-expire. (DD-144)
——— BRD-2 AUDIT AMENDMENTS 2026-05-13 ———
New FRs added to address audit-identified gaps.
- FR-TRUST-5 — /coherence:trust --status output. Renders Markdown report with: (a) current trust state (promoted? auto-land kinds? promote_hint_emitted_at?), (b) top 5 highest-scored personal sections, (c) top 5 lowest-scored personal sections, (d) team aggregate summary (active developer count, contested-section count), (e) promote eligibility check (which threshold conditions met / pending). Always-on, no LLM.
- FR-LEDGER-5 — First-run / re-install state initialization. When .claude/coherence/trust-ledger.json is absent, treat as empty ledger (no sections scored, no events). New sections start at score 0.0 (FR-TRUST-1 carry). On plugin re-install (DD-118), existing trust-ledger.json is preserved unchanged. When coherence/trust/\<own-hash\>.json is absent (fresh repo or first sync), /coherence:trust sync creates it with the developer's current personal scores plus last_synced_at.
- FR-ASSERTS-5 — Unknown / deferred assertion type behavior. If a section declares asserts: with a type not in the v1.0 supported set (\{has_example, no_placeholder_links, max_words, min_words, no_todo_comments, symbol_exists, file_exists\}), the assertion is logged and IGNORED (validation proceeds). This permits forward-compatible authoring (e.g. v1.0 sections can declare signature_matches in anticipation of v1.1 without breaking v1.0 validation).
- FR-MANIFEST-5 — New slash commands registered in .claude-plugin/plugin.json#slashCommands (DD-130 autogen carry). v1.0 adds: coherence:trust (with subcommands as -- flags), coherence:metrics. /coherence:audit and /coherence:repair already present in v0.4 manifest. scripts/generate-command-stubs.mjs regenerates commands/\<name\>.md stubs at build time per FR-AUTOGEN-1 carry.
- FR-TELEMETRY-1 — Trust ledger events. Two new metrics.jsonl event types: proposal_accept_recorded (emitted on accept event recording in trust ledger), proposal_revert_recorded (emitted on revert). Both follow v0.3 DD-068 redaction matrix: sectionRef hashed if upload consent is opt-out (default), full sectionRef if upload consent is opt-in. No clear-text section content. No author email. Schema fields: _ts, event_type, section_ref_hash (or section_ref), weight, author_hash.
——— BRD-2 AUDIT PASS 3 AMENDMENTS 2026-05-13 ———
- FR-LEDGER-4 amendment (pass 3) — Contested section threshold. A section’s team aggregate is FLAGGED as contested when \|aggregate\| \< 0.2 (operational threshold from M-LEDGER-4). Flag surfaces in: (a) /coherence:metrics Top drifting sections section with a ⚠ marker, (b) /coherence:trust --status team aggregate summary (contested-section count). Threshold is hardcoded in v1.0; configurable in v1.1 via coherence/config.json#trust.contested_threshold.
- FR-ASSERTS-1 amendment (pass 3) — Logging channel for ignored assertions. Both unknown-type (FR-ASSERTS-5) and excess-count (DD-143 pass 2 amendment, max 10 per section) ignored assertions are logged to stderr at session start with a single combined warning line per section: “coherence: \<count\> assertion(s) in section \<ref\> ignored: \<reasons\>”. Not emitted to metrics.jsonl (not a privacy-sensitive event). Not logged to coherence-log.md (not a state-changing event). One warning per session per section even if multiple problematic assertions exist.
- FR-TELEMETRY-1 amendment (pass 3) — weight field semantics. The weight field in proposal_accept_recorded / proposal_revert_recorded events is the RAW event weight from DD-138 amendment pass 2: +1.0 for accept, 0.0 for edit (proposal_edit_recorded event — implicit), −1.0 for revert. NOT the decayed contribution (which depends on time and is computed at score-read time, not write time). Edit events use a separate proposal_edit_recorded event type with weight 0.0 — retained for denominator-evidence weighting (DD-138 pass 2).
