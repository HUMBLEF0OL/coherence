**Source corpus: **[v1.0](https://www.notion.so/35f010d46a7081f39e22c469bc63c01b) (substrate, shipped 2026-05-13). v1.0.0 release tag pinned at [5a9bf57](https://github.com/HUMBLEF0OL/coherence/releases/tag/v1.0.0). Findings sourced from the mcp-sentry smoke run against branch [test/coherence-v1-smoke](https://github.com/HUMBLEF0OL/mcp-sentry/tree/test/coherence-v1-smoke).
*v1.0 shipped under contention — every milestone closed and the release was cosign-signed and Rekor-verified, but six bugs were caught in post-tag audits (one of them, the consent.ts plugin_version constant, would have written a stale '0.4.0' into every fresh v1.0 install). v1.0.1 books those fixes formally, hardens the release pipeline so the same class of misses cannot ship again, and closes the LLM-stage validation gap that the deterministic smoke could not reach.*
**Theme**
**v0.1 reacts. v0.2 proposes. v0.3 distributes. v0.4 polishes. v1.0 trusts. v1.0.1 verifies.**
v1.0.1 is the verification + hardening release. It rolls up six post-tag fixes, closes the gap between "the deterministic detection layers pass against a real consumer" (proven by the mcp-sentry smoke) and "the LLM-driven Stage 1 + Stage 2 + deep-audit stages produce useful patches against real drift" (still unproven), and rewrites the install story so downstream consumers can actually register the plugin into Claude Code instead of just laying down state files.
**Problem statement**
v1.0 shipped cleanly per the spec, but a post-tag audit + downstream smoke surfaced four classes of problem that need v1.0.1 closure:
- **Six bugs caught post-tag, not by pre-release tests. **README cosign verification regex render bug (the \$ in tags/v.\* was interpreted as a backreference token); cost-trend reading _ts instead of the schema's \`timestamp\` field; in-process trust-ledger mutex cleanup branch never firing (slow memory leak per ledger path); DEFAULT_PLUGIN_VERSION constant hardcoded to '0.4.0' in consent.ts (assertVersionSync didn't scan embedded constants); .gitignore patch covering only signal-cache.json + session-map.json (NFR-PRIVACY-N5 leak risk for trust-ledger.json, metrics.jsonl, cost-ledger.json, coherence-log.md, scan-cache/, proposal-cache.json); the M-LEDGER-1 atomic-write test was sequential, never exercising the actual concurrent read-modify-write race. Each fix is already on master — v1.0.1 books them in a CHANGELOG entry and ensures each has a regression test that would have caught it.
- **LLM stages unproven against a real consumer. **The mcp-sentry smoke (test/coherence-v1-smoke fixture branch) exercised the deterministic detection + validation layers (anchor scan, section index, asserts pipeline, token-budget classifier, hallucination grep) end-to-end on a real TypeScript monorepo. Stage 1 + Stage 2 + /coherence:audit --deep never ran because they require an interactive Claude Code session. Without cassette recordings, the LLM path is unverified outside the v1.0 unit-tested code paths.
- **Install story has a documentation gap. **runFreshInstall (the firstRun bootstrap) lays down .claude/coherence/ state but does NOT register the plugin into Claude Code's plugin system. State + no registered hooks = inert plugin. Until cohrence is published to the official Anthropic marketplace, downstream installs require manual settings.local.json or \~/.claude/settings.json edits that today's README doesn't explain.
- **Release pipeline doesn't enforce its own invariants. **assertVersionSync only checks three top-level version sources (package.json, .claude-plugin/plugin.json, src/state/init.ts PLUGIN_VERSION). It does not scan src/ for embedded version constants, which is exactly the class of bug that produced fix #4 above. Concurrency tests only exist for the trust ledger — not the other RMW state files (proposal-cache, signal-cache, scope-cache).
**Goals (v1.0.1)**
- **G1 — Six post-tag fixes booked. **Every fix on master between cb52271..1ca029a is enumerated in RELEASE_NOTES_v1.0.1.md with its commit ref, the bug class, the fix, and the regression test that catches it. Suite passes at ≥ 1000 tests (currently 977; +cassette coverage).
- **G2 — Release pipeline can catch the next plugin_version regression. **assertVersionSync extended to scan src/\*\*/\*.ts for /\['"\]\\d+\\.\\d+\\.\\d+\['"\]/ in identifier contexts containing 'version', surfacing any deviation as a release-blocking error. Plus a CI-time test that introduces a deliberate stale constant in a fixture file and asserts the gate trips.
- **G3 — LLM stages validated against a real downstream consumer. **Cassettes recorded for Stage 1, Stage 2, and /coherence:audit --deep against the mcp-sentry test/coherence-v1-smoke fixture. New tests/e2e/mcp-sentry-fixture.test.ts asserts the full Stop pipeline produces patches against the drifted symbols. The trust ladder is exercised with synthetic 30-day session history covering FR-TRUST-4 promote eligibility + auto-accept sweep.
- **G4 — Install story is unambiguous. **README ## Install rewritten with three documented paths (marketplace, local-path settings.local.json, global \~/.claude/settings.json). Post-install verification is /coherence:status. Cassette-refresh path documented for downstream consumers. cohrence submitted to the Anthropic plugin marketplace (M-LISTING-1 gate).
**Non-goals (post-v1.0.1)**
- Hosted backend, database, or upload service — ever (DD-117 permanent).
- Cross-major migration logic (DD-118 permanent). The v1.0 → v1.0.1 patch path is in-place; no migrator.
- Trust ladder field calibration against real downstream telemetry — deferred until ≥ 50 sessions × ≥ 30 days observed in distributed installs (DD-116 / TC-2).
- Live LLM call orchestration for the cassette-refresh workflow on downstream consumer projects — cassettes remain in the coherence repo only; downstream installs always hit the production Anthropic API path.
